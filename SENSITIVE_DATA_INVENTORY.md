## 5. FIRESTORE SECURITY RULES

### File: `firestore.rules`
- ❌ Currently OPEN (not in `.gitignore`)
- ✅ SAFE to commit (contains no secrets, rules-based access)

### Current Rules:
```
- Test mode (allow all) for development
- Should be restricted for production
```

### Production Rules (example):
```firestore
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /invoice_sends/{document=**} {
      allow read: if request.auth != null;
      allow write: if false;
    }
    match /scheduled_invoices/{document=**} {
      allow read, write: if request.auth != null;
    }
  }
}
```

## 8. API KEYS (PUBLIC vs PRIVATE)

### Firebase Public API Key:
- **Status**: Safe to expose (Firebase restricts by rules)
- **Why**: Restricted via Firestore security rules
- **Stored in**: `.env` as `VITE_FIREBASE_API_KEY`

### Private Keys (Service Account):
- **Status**: ❌ NEVER expose
- **If needed**: Generate from Firebase Console
- **Location**: `.env` > Never in code
- **Use**: Backend operations only

### Rule of Thumb:
```
Public Key (Vite/Frontend) → OK to expose, restricted by rules
Private Key (Backend) → NEVER expose, used by server
```

## 10. GIT HOOKS PROTECTION

### What Gets Blocked:
✅ Added file `.gitleaks.toml` with patterns for:
- Firebase API Keys (regex: `AIzaSy[a-zA-Z0-9\-_]{35}`)
- AWS Access Keys (regex: `AKIA[0-9A-Z]{16}`)
- Private Keys (regex: `-----BEGIN.*PRIVATE KEY`)
- Passwords (`password\s*[:=]`)
- API Keys (`api_key\s*[:=]`)
- Secrets (`secret\s*[:=]`)
- Specific credentials (like known Gmail password)

### Pre-Commit Hook:
```bash
.githooks/pre-commit
- Runs on each `git commit`
- Scans staged files for patterns
- Blocks commit if secrets detected
- Shows which file and which line

Error message example:
❌ ERROR: Sensitive file detected: .env
⚠️  COMMIT BLOCKED: Potential secrets detected!
```

### Pre-Push Hook:
```bash
.githooks/pre-push
- Runs on each `git push`
- Uses gitleaks if installed
- Prevents pushing commits with secrets
- Can be bypassed with --no-verify (NOT RECOMMENDED)
```

## 11. TEAM ONBOARDING

### Checklist for New Team Members:
1. ✅ Clone repository
2. ✅ Run: `bash setup-security.sh` (or `setup-security.bat` on Windows)
3. ✅ Copy: `cp africanite/.env.example africanite/.env`
4. ✅ Edit: Add actual Firebase credentials to `.env`
5. ✅ Download: `google-services.json` from Firebase Console
6. ✅ Install: `npm install` in both `africanite/` and `functions/`
7. ✅ Test: `npm run dev` should work

### What NOT to Do:
- ❌ Share `.env` files via Slack/Email
- ❌ Commit `.env` to git
- ❌ Hardcode API keys in code
- ❌ Use `git commit --no-verify` to bypass hooks
- ❌ Push branches with unreviewed secrets

### If You Commit a Secret By Accident:
1. STOP immediately
2. Regenerate the API key in Firebase Console
3. Remove from local commits: `git reset HEAD~1`
4. Remove from file: `git checkout -- FILE_NAME`
5. Update `.env` with new key
6. Commit clean version

## 🚨 EMERGENCY: Revoked Credentials

If Firebase API Key or Gmail password is compromised:

1. **Invalidate immediately**:
   ```bash
   # Firebase Console → Project Settings → API Keys
   # Delete or regenerate the exposed key
   
   # Gmail → Manage Your Google Account → Security
   # Remove the app-specific password
   ```

2. **Update locally**:
   ```bash
   # Update .env
   VITE_FIREBASE_API_KEY=<NEW_KEY>
   
   # Update Firebase config
   firebase functions:config:set gmail.password="<NEW_PASSWORD>"
   ```

3. **Force all deployments**:
   ```bash
   firebase deploy --force
   npm run build && npm run deploy # For frontend
   ```

4. **Clean history** (if pushed):
   ```bash
   bfg --delete-files .env
   git reflog expire --expire=now --all
   git gc --prune=now
   ```
